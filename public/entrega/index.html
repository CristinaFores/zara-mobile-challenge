Estado en la URL — qué parámetro hace qué

Nada de configuración vive solo en memoria del cliente: todo lo compartible va en la ruta o en query params.

Dónde Parámetro Para qué sirve
/ ?search= Filtra el catálogo contra la API (no solo en cliente). Debounce de 300 ms. El enlace es compartible y atrás/adelante restaura la búsqueda.
/products/[id] [id] en la ruta Identifica el móvil concreto. Si no existe en la API, la app responde con página 404.
/products/[id] ?color= Nombre del color elegido. Cambia la imagen del hero y forma parte de la clave al añadir al carrito. Actualiza la URL con history.replaceState (sin refetch del RSC). Sin valor válido, el botón de compra sigue deshabilitado.
/products/[id] ?storage= Capacidad elegida (p. ej. 256 GB). Fija el precio mostrado y la variante que se guarda en el carrito. Actualiza la URL con history.replaceState (sin refetch del RSC). Sin valor válido, el botón de compra sigue deshabilitado.
/cart — El carrito no usa query params: persiste en localStorage. Cada línea enlaza de vuelta al detalle con los mismos color y storage.

Imágenes optimizadas en servidor

Las fotos de la API llegan grandes y con fondo blanco. El navegador nunca pide la imagen cruda con credenciales.

  • Proxy /api/images. El cliente pide /api/images?url=…&w=…&q=…; el servidor descarga la foto, la procesa y devuelve WebP con cabeceras de caché.
  • Recorte de fondo con Sharp. Flood-fill desde los bordes: el blanco exterior pasa a transparente; el blanco interior (pantalla del móvil) se conserva.
  • Redimensionado y caché. Ancho máximo 1200 px, calidad configurable y caché en memoria (~200 entradas) para no repetir trabajo Sharp en cada visita.
  • Seguridad. Solo se aceptan URLs del host de la API del reto; protocolo http/https. Cubierto en imageProcessing.test.ts y api/images/route.test.ts.

Carrito — identidad, persistencia y precios

Cada línea es una configuración completa (móvil + color + almacenamiento), no solo el id del producto.

Sync de precios

Reconciliación preparada

Existe syncPrices(mapa) en el reducer: actualiza precios por clave y recalcula el total sin tocar color, imagen ni nombre. Testeado; listo para conectar si la API expusiera cambios de precio en vivo.

Identidad

Clave única por variante

Clave id::color::capacidad. El mismo móvil en dos colores o dos memorias genera dos líneas distintas. Repetir la misma configuración no duplica la línea.

Persistencia

localStorage seguro

Hidratación solo en cliente. Al leer, se valida el JSON y se descartan entradas corruptas. Tests en cartStorage.test.ts y E2E de recarga en e2e/cart.spec.ts.

Precio al añadir

Snapshot en el momento de compra

Al pulsar añadir, la línea guarda el precio de la variante elegida. El total del carrito se recalcula desde esas líneas. Cubierto en CartContext.test.tsx.

Producto inexistente

404 en detalle

Un id inválido en /products/[id] dispara notFound(). La API del reto no expone stock: el carrito no elimina líneas automáticamente si un producto desaparece del catálogo.

Tests E2E

Flujos reales

Playwright cubre carrito vacío, añadir con configuración, ver total, eliminar línea y persistencia tras recargar la página.

Criterios

Resumen de cumplimiento del enunciado. El detalle técnico está en el README y en AGENTS.md.

Catálogo

Listado y búsqueda

Rejilla de 20 productos, búsqueda contra la API con debounce, contador de resultados, estados de carga/error/vacío y navegación al detalle.

Detalle

Configuración y similares

Imagen principal, selectores de color y almacenamiento, precio según variante, especificaciones, productos similares y botón deshabilitado hasta elegir ambas opciones.

Carrito

Persistencia

Guardado en localStorage sin romper el SSR, identidad por producto + color + almacenamiento, eliminar líneas, total, contador en cabecera y volver al catálogo.

API

Seguridad

La clave API_KEY solo existe en servidor. Las peticiones con credenciales pasan por route handlers o server components, nunca por variables NEXT_PUBLIC_.

UI

Visual y responsive

Sass + BEM + CSS Modules, Helvetica, bordes de 1 px, diseño mobile-first, next/image y tokens en _variables.scss.

Entrega

Calidad y documentación

TypeScript strict, HTML semántico, metadata SEO, consola sin errores, README bilingüe y esta página de entrega con demo embebida.

Decisiones
  • Next.js App Router frente a una SPA pura. Renderizado en servidor para catálogo y detalle, metadata dinámica por producto, route handlers para credenciales e imágenes optimizadas con next/image.
  • Estado en la URL, no en un store global. Búsqueda (?search=) navega con el router de Next. Color y almacenamiento (?color=, ?storage=) actualizan la barra de direcciones con history.replaceState para enlaces profundos, atrás/adelante y refresh sin volver a ejecutar el RSC ni remontar imágenes del detalle.
  • Optimización de assets con proxy Sharp. Las imágenes de la API no tenían tamaños ni crops consistentes. El route handler /api/images normaliza fondo y encuadre en servidor, sirve WebP con caché y evita upscale agresivo. La clave de la API no llega al navegador.
  • Contrato visual desde Figma vía MCP. El archivo del reto se leyó con MCP Figma; medidas, paddings, gaps, tipografía y frames de referencia se consolidaron en DESIGN.md y en los tokens de src/scss/_variables.scss.
  • Motion fuera del mínimo funcional. Transición visual catálogo → detalle con View Transitions API, reorder del grid al buscar con FLIP y fundido entre capas de color precargadas en el hero. Todo respeta prefers-reduced-motion.
  • Validación visual de entrega. Capturas en public/entrega/screenshots/ tomadas sobre el deploy real para contrastar catálogo, detalle, carrito y mobile con el Figma del challenge.
  • Context API solo para el carrito. Sin Redux ni TanStack Query: el carrito es el único estado compartido en cliente; el resto es server-first o depende de la URL.
  • Calidad integrada en el flujo de trabajo. CI en cada PR, SonarCloud, tests con simulación HTTP y E2E con Playwright. El README actúa como contrato de revisión, no como anexo posterior.
